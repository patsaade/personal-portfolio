Szechuan Sauce is one of the better full-scope DFIR challenges out there — disk, memory, and packet captures across multiple hosts. This writeup focuses on the **memory analysis** track: what I pulled from the RAM image and how it tied into the broader intrusion story. Flags are intentionally omitted; the methodology is the point.

## Challenge Summary

A small environment — a domain controller and a couple of workstations — gets compromised. You're handed triage collections including memory images, and asked to reconstruct initial access, the malware involved, lateral movement, and the indicators. The memory image I focus on here is the workstation dump.

## Approach

My opening sequence was the standard first-30-minutes pass from my [Volatility install guide](/blog/volatility3-installation):

```bash
vol -f workstation.raw windows.info
vol -f workstation.raw windows.pstree
vol -f workstation.raw windows.cmdline
vol -f workstation.raw windows.netscan
vol -f workstation.raw windows.malfind
```

`pstree` was the first tell. There was a process parented in a way that didn't fit normal Windows startup — a service-like binary running from a non-standard path, with a child shell.

## Key Findings

### 1. Suspicious process & path

`pslist`/`psscan` agreed (no DKOM hiding here), but the **image path** was the giveaway — an executable living outside `System32` masquerading as a system component. `cmdline` showed it launched with arguments consistent with a remote-access tool, not a legitimate service.

```bash
vol -f workstation.raw windows.cmdline --pid <suspect_pid>
```

### 2. Network beacon

`netscan` tied that PID to an outbound connection to an external IP on a non-standard port — the C2 channel. Correlating the PID across `netscan` and `pstree` is what turned "weird process" into "this is the foothold."

### 3. Remote access tooling

The artifact pattern — service install, outbound connection, interactive shell child — matched a known open-source RAT. Dumping the process region for static triage confirmed strings consistent with that family.

```bash
vol -f workstation.raw windows.pslist --pid <suspect_pid> --dump
```

### 4. Lateral movement evidence

Command-line history and handles pointed at credential access and remote execution against the DC — consistent with the intrusion moving from the workstation toward domain assets. This is where the memory track hands off to the disk/registry track (service creation events, prefetch of remote-exec tooling).

## Timeline

| Time (rel.) | Event | Source artifact |
| --- | --- | --- |
| T0 | Malicious binary executed from non-standard path | `pstree`, `cmdline` |
| T0 + min | Outbound C2 connection established | `netscan` |
| T0 + min | Interactive shell spawned as child | `pstree` |
| T0 + hrs | Credential access / remote exec toward DC | handles, cmdline |

## IOCs Extracted

- Malicious binary name + on-disk path (non-`System32`)
- SHA-256 of the dumped process image
- C2 IP address and destination port
- RAT family identification (string/heuristic match)
- Service name used for persistence
- Suspicious parent/child process pair
- Outbound connection PID linkage

## Lessons Learned

- **`pstree` before everything.** The entire case unspooled from one out-of-place parent/child relationship. Read process trees as narratives.
- **PID is the join key.** The moment you can tie one PID across `pstree`, `cmdline`, and `netscan`, a loose suspicion becomes a defensible finding.
- **Memory is the pivot, not the whole story.** The RAM image told me *what was running and talking*; the disk and registry told me *how it survived and where it went*. Multi-source challenges reward correlating, not siloing.

A genuinely fun challenge — and a good reminder that the boring opening sequence is what cracks even the hard ones.
